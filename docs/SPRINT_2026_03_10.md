### Critical Observation: Upset Column Behavior

The `upset=True` column records games where the model-predicted loser (underdog by ELO) actually
won. The model accuracy on upsets is exactly 0% because the column is defined as "we predicted
the favorite, but the underdog won" — that is the definition of a prediction error. This is not
a bug. It means: the model correctly identifies favorites, but cannot predict which upsets will
happen. Upset rate in this dataset is 75/322 = 23.3%, which is consistent with historical NBA
upset frequency.

The actual question is: does the model produce upsets at the right rate? Our model predicts the
stronger-ELO team 65%+ of the time, matching historical win rates. This is correct behavior.

### 50–100 ELO Diff Anomaly

band. This is the primary mechanical bug to investigate this sprint.

## Priority Fix 1 — Diagnose and Fix 50–100 ELO Diff Bucket (Expected Gain: +3–5% in segment)

### What is happening

Games with an ELO difference of 50–100 points should produce win probabilities in the
58–63% range for the stronger team. The model produces 52.46% accuracy on 61 such games.
This means the model is essentially treating 50–100-point advantages as coin flips.

Three candidate causes:

**Cause A: Close game enhancer is misfiring in this band.** The `CloseGameEnhancer` activates
when `abs(elo_diff) < 100` (line 22 of `close_game_enhancer.py`: `CLOSE_GAME_THRESHOLD = 100`).
That means almost the entire 50–100 bucket falls inside close-game mode, which overrides the
base probability with H2H and travel adjustments that may be noisy at this sample size.

**Cause B: The final probability recalculation at line 701 of `hybrid_team_player.py` uses
`home_advantage=0`.** For games in the 50–100 range where the away team is the stronger team,
the missing home advantage (60 ELO points) could flip a predicted away win into a true coin
flip. If `home_advantage` were correctly applied, some of these would become correct picks.
The comment at line 698 reads "HA is captured implicitly via WElo" — but WElo adjustments are
dampened in tossup games (< 30 ELO diff) and may be insufficient in the 50–100 range.

**Cause C: The tossup compression at lines 715–725 extends to `abs(raw_hybrid_diff) < 30` only,
but the close game enhancer's threshold is 100.** There is a 30–100 ELO-diff band where close
game enhancement fires (reducing the signal from ELO differences) but tossup compression does
not fire. This leaves games in the 50–100 range with degraded signal and no offsetting correction.

### Investigation steps

1. Isolate the 61 games where `abs(elo_diff)` is between 50 and 100 in
   `data/exports/prediction_tracking.csv`. Check what fraction have `is_close_game=True`,
   `close_game_enhancement_active=True`, and whether the model predicted home or away.

2. In `src/predictors/hybrid_team_player.py` lines 539–544, add a temporary debug log:
   ```python
   logger.debug(f"[50-100 BAND] raw_hybrid_diff={home_hybrid - away_hybrid:.1f}, "
                f"home_advantage=60 would give prob={calculate_win_probability(home_hybrid, away_hybrid, 60):.3f}, "
                f"current base prob={home_win_prob:.3f}")
   ```
   Run this across 10 games where `abs(elo_diff)` is 50–100 and compare the probabilities.

3. Test whether raising `CLOSE_GAME_THRESHOLD` from 100 to 50 in `close_game_enhancer.py`
   improves the 50–100 bucket without degrading the <50 bucket. This would exclude the 50–100
   range from close-game mode entirely, letting ELO signal dominate.

4. Separately, confirm the home advantage bug: at line 701, change `0` to `60` for a one-day
   backtest on the 61 games in the 50–100 bucket. If this raises bucket accuracy above 58%,
   the home advantage bug (Sprint 03-09 Item 4) is the root cause.

### Concrete code change candidates

**Option A — Narrow the close game threshold:**
In `src/features/close_game_enhancer.py` line 22:
```python
# Current:
CLOSE_GAME_THRESHOLD = 100
# Candidate:
CLOSE_GAME_THRESHOLD = 50
```
This removes 50–100 diff games from close-game mode. Backtest required before deploying.

**Option B — Fix the home advantage passthrough (Sprint 03-09 Item 4, concrete form):**
In `src/predictors/hybrid_team_player.py` line 701:
```python
# Current (with incorrect comment):
final_home_prob = calculate_win_probability(
    final_home_elo, final_away_elo, 0  # Home advantage already applied
)
# Candidate:
final_home_prob = calculate_win_probability(
    final_home_elo, final_away_elo, home_advantage
)
```
The comment at line 698–700 argues WElo covers home advantage implicitly. But WElo is
dampened in tossup games. Outside tossup range (>30 diff), WElo may or may not fully
cover 60 ELO points. This must be backtested, not assumed.

### Backtest requirement

Before deploying either change:
1. Capture baseline on current 322-game dataset: segment accuracy by ELO bucket
2. Apply change to a copy of `hybrid_team_player.py`
3. Re-run predictions on the same 322 games using the modified logic

## Priority Fix 2 — Resolve Home Advantage Fragmentation (Expected Gain: +1–2% overall)

### What is happening

The home advantage value is fragmented across the codebase with four different values in use:

| Location | Value | Used For |
|----------|-------|----------|
| `config/settings.yaml` line 20 | 60 | Config (not read by engines) |
| `src/predictors/hybrid_team_player.py` (implicit) | 60 | Main prediction path |
| `src/predictors/hybrid_team_player.py` line 701 | 0 | Final probability recalc |
| `src/engines/hybrid_predictor.py` line 58 | 30 | Hybrid predictor (legacy) |
| `src/engines/team_elo_engine.py` line 32 | 20 | ELO training updates |
| `process_game_elo_update()` in `elo_math.py` | 30 | Rating updates |

The prediction path (`hybrid_team_player.py`) applies home advantage (60 points) at the
initial base probability on line 540–544 but then drops it at line 701 for the final
recalculation. Every adjustment applied between lines 544 and 701 (rest, WElo, injury) is
calculated on raw ELO values that do not carry the home advantage forward.

This means: home advantage is applied, then discarded, then not re-applied. The final
probability for close games is systematically biased toward 50% regardless of home court.

The home accuracy of 69.44% vs away accuracy of 61.27% already shows the model under-weights
home court in away predictions. Fixing this should improve the 61.27% away bucket.

### Concrete code change

**Step 1: Confirm the bug magnitude.**
Add a log line at `hybrid_team_player.py` line 700 (before the final `calculate_win_probability`
call) that computes:
```python
prob_with_ha = calculate_win_probability(final_home_elo, final_away_elo, home_advantage)
prob_without_ha = calculate_win_probability(final_home_elo, final_away_elo, 0)
logger.info(f"HA audit: with={prob_with_ha:.3f}, without={prob_without_ha:.3f}, "
            f"delta={prob_with_ha - prob_without_ha:.3f}")
```
Run this on 5 games. If the delta is consistently near 0 (WElo is covering it), the bug
is minor. If delta is 3–8 percentage points, the bug is significant.

**Step 2: Fix if significant.**
Change line 701 from `home_advantage=0` to `home_advantage=home_advantage`. Remove the
misleading comment. Backtest against all 322 tracked games.

**Step 3: Consolidate the config.**
`config/settings.yaml` documents that the config value is not read by engines. Fix this:
In `hybrid_team_player.py`, read home_advantage from settings.yaml via `load_settings()`.
Set the YAML value to 60. This ensures a single source of truth.

**Step 4: Document the decision on ELO training vs prediction home advantage.**
The ELO update in `team_elo_engine.py` uses 20 points, while prediction uses 60. This is
intentional and correct — lower home advantage in training prevents overfitting, while
prediction uses the empirical 60-point calibration. Add a comment in both files explaining
this distinction.

### Backtest requirement

- Before: capture overall accuracy, predicted-home accuracy, predicted-away accuracy
- After: same three metrics on the same 322-game set
- Success threshold: predicted-away accuracy improves (target: 63%+), overall holds at 65%+
- Save to `data/backtest_results/2026-03-10-home-advantage-fix.csv`

## Priority Fix 3 — Implement Season Reset Reversion (Expected Gain: +1–2% early-season)

### What is happening

The model carries ELO ratings forward without reversion at the season boundary. This violates
a well-established principle from FiveThirtyEight's methodology: teams regress toward the mean
between seasons because rosters change, coaches change, and the sample of prior-season results
has declining relevance.

Currently: `team_elo_engine.py` loads prior ratings directly with no reversion.

Without reversion, teams that were extreme at end-of-season (e.g., OKC at ~1770 or Utah at
~1280) carry those extremes into the new season at full weight. Over 20–30 games, the model
corrects, but it costs accuracy during that correction window.

The 2025-26 season started on Oct 1, 2025. The model has been running since Nov 30. The
early-season correction window (Oct–Nov) has passed, so this fix will primarily benefit
the 2026-27 season. However, it must be implemented before the 2026 playoffs begin so
the off-season logic is ready.

### Concrete implementation

In `src/engines/team_elo_engine.py`, add a `apply_season_reset()` method:

```python
def apply_season_reset(self, prior_season_end_date: str, league_mean: float = 1500) -> None:
    """
    Apply FiveThirtyEight-style preseason reversion.
    Formula: preseason_rating = 0.75 * prior_end_rating + 0.25 * league_mean
    Call this once, after loading prior season ratings, before processing new season games.
    """
    for team_id in self.current_ratings:
        prior = self.current_ratings[team_id]
        reverted = 0.75 * prior + 0.25 * league_mean
        logger.info(f"Season reset {team_id}: {prior:.1f} -> {reverted:.1f} "
                    f"(delta: {reverted - prior:+.1f})")
        self.current_ratings[team_id] = reverted
```

Add season boundary detection in `scripts/daily_update.py`: detect when current date >= Oct 1
and prior ratings are from the previous season (prior season end = Jun 30). Call
`apply_season_reset()` before running the first game of the new season.

### Backtest requirement

This change cannot be evaluated against the current 322-game live prediction set (those games
are mid-season). The correct backtest is against historical seasons:

1. Run `scripts/backtest_2024_25_season.py` with and without season reset applied at Oct 1
2. Compare Nov accuracy (games 1–20 of season) specifically — that is when the benefit shows
3. Also run full historical backtest (31,843 games) if compute allows
4. Save to `data/backtest_results/2026-03-10-season-reset.csv`

If the historical backtest is not feasible before playoffs, defer deployment to June 2026
(between seasons). Do not deploy to production mid-season without full backtest.

### Fix #2 — Home Advantage Bug (Skipped — Already Investigated)

**File in question:** `src/predictors/hybrid_team_player.py` lines 593 and 701

**Outcome:** Not deployed. Investigation was completed 2026-03-09 as Sprint 03-09 Item 4.
